# Learning Loop System ‚Äî Design

Date: 2026-06-01
Status: design (pre-implementation)
Feeds: spec `0006-learning-loop-system`

## Why

The Daily Command Center got better when it stopped being an isolated routine and
became the read-surface of a closed loop. Capture, classification, durable state,
the daily view, an integrity check, and weekly curation are six parts of one system ‚Äî
not six unrelated briefs. This document is the validated design for that loop.

| # | Component | Role |
| --- | --- | --- |
| 01 | **Granola** | Captures meetings: objections, discoveries, follow-ups. Intake only. |
| 02 | **Learning Loop** | Classifies each signal as learning / hypothesis / decision (or routes a task). |
| 03 | **_central** | Holds the operational state that must survive the chat. Source of truth. |
| 04 | **DCC** | Shows only what changes *today*. |
| 05 | **Watchdog** | Checks that routines are fresh and recording correctly. |
| 06 | **Weekly Review** | Promotes, demotes, archives, or corrects learnings. |

## Invariants (carried from the engine)

1. **Routines never write.** Every routine runs `write_allowed: false`. The only mutation
   path is the explicit `apply` command, which lives *outside* the run path.
2. **Observed is not persisted.** A routine proposes; a human (or an explicit `apply`)
   persists. "Hypothesis ‚âÝ truth" ‚Äî uncertain signals enter as `candidate`, never
   `promoted` by the thing that proposed them.
3. **git is the audit log.** State is git-tracked; every `apply` is a stageable diff.
   We do not build a second audit system inside the files.
4. **_central is the source of truth; Linear is a mirror.** When they disagree,
   `_central` wins and the gap is surfaced as a Problem.

## State store: `_central/state/`

A typed, git-tracked store ‚Äî the single thing read and (via `apply`) written across
routines.

### `learnings.jsonl` ‚Äî mutable record set (one JSON object per line)

Knowledge only. `kind:task` does **not** live here (see Task routing).

```jsonc
{
  "id": "lrn_2026-06-01_a1b2",
  "created": "2026-06-01T08:40:00Z",
  "kind": "learning|hypothesis|decision",
  "status": "...",                 // per-kind, schema-enforced (table below)
  "text": "one short sentence",
  "origin": "granola:meeting/acme-2026-06-01",
  "supersedes": null,              // or the id this record corrects
  "last_transition": {
    "at": "2026-06-01T08:40:00Z",
    "actor": "weekly-review|apply|learning-loop",
    "reason": "observed in 5 customer calls",
    "op_id": "op_456"
  }
}
```

**Mutable, not append-only.** Status flips happen in place; full per-record history is
recovered from `git log -p state/learnings.jsonl`. This is a deliberate choice (Option A):
simplicity over event-sourcing. We do not claim "append-only" anywhere.

**Per-kind status (enforced by JSON Schema `if kind then status-enum`):**

| kind | allowed status |
| --- | --- |
| `learning` | candidate ¬∑ promoted ¬∑ archived ¬∑ rejected |
| `hypothesis` | candidate ¬∑ promoted ¬∑ archived ¬∑ rejected |
| `decision` | active ¬∑ archived |

A *rejection* is not a kind ‚Äî it is `status: rejected` on a learning/hypothesis.

**Corrections never edit text in place.** A correction writes a **new** record with
`supersedes: <old_id>` and sets the old record to `archived`, preserving history.

### `decisions.md` ‚Äî generated projection

A human-readable view of `kind:decision` records, **regenerated by `apply`** ‚Äî never
authoritative, never written independently. The JSONL is the system of record; a decision
can never exist only in the markdown.

### `index.yaml` ‚Äî freshness & health

```yaml
routines:
  daily-command-center: { last_run: 2026-06-01T08:40Z, sla_hours: 26, status: fresh }
  learning-loop:        { last_run: 2026-06-01T08:35Z, sla_hours: 26, status: fresh }
counts: { candidate: 12, promoted: 30, archived: 7, rejected: 4 }
schema_version: 1
```

Watchdog reads this and proposes `reindex` corrections; `apply` recomputes it.

## The `apply` command ‚Äî the only writer

Routines emit, in their existing `markdown+json` block, a `proposals[]` array:

```jsonc
{ "op": "add",        "record": { /* learning.json minus id/created */ } }
{ "op": "transition", "id": "lrn_...", "to": "promoted", "reason": "..." }
{ "op": "correct",    "id": "lrn_...", "text": "...",     "reason": "..." }
{ "op": "task",       "title": "...", "origin": "granola:...", "target": "linear" }
{ "op": "reindex" }
```

`run.ts apply <routine> [--entry N] [--op M] [--all] [--dry-run]`:

1. Read the routine's latest proposals (from its `memory.md` run record / saved artifact).
2. Validate each resulting record against `state/schema/learning.json`.
3. Apply:
   - `add` ‚Üí stamp `id`+`created`, append to `learnings.jsonl`.
   - `transition` ‚Üí flip `status` + `last_transition` (with a fresh `op_id`).
   - `correct` ‚Üí write new record `supersedes` old; archive old.
   - `task` ‚Üí route to the **Linear mirror** (create the issue); never touches `learnings.jsonl`.
   - `reindex` ‚Üí recompute `index.yaml` and regenerate `decisions.md`.
4. Print a diff. `--dry-run` prints without writing.

`apply` writes only under `state/` (paths outside are refused). It never auto-commits.

## Engine changes (minimal)

- **`core/state.ts`** (new) ‚Äî pure read/append/transition/correct + `index.yaml` recompute
  + `decisions.md` projection. No I/O in the core logic (strings in / strings out), mirroring
  `core/memory.ts`.
- **`cmdApply` in `run.ts`** (new) ‚Äî the I/O shell around `core/state.ts`.
- **No change** to `contract`, `compose`, `validator`, `runner`, `preflight`, or `guards`.
  `apply` is outside the run path, so `write_allowed: false` still governs every run.

## Routines (pure artifacts, all `write_allowed: false`)

- **`granola-capture`** *(new)* ‚Äî reads Granola notes (via its Gmail meeting-reports + Drive,
  since there is no Granola MCP). Output: raw captured items (objection / discovery /
  follow-up / decision-mentioned), each with `origin`. No classification.
- **`learning-loop`** *(new)* ‚Äî reads `granola-capture` output + Gmail/calendar context +
  recent `state/`. Output: `proposals[]` classifying signals as `learning|hypothesis|decision`,
  plus `{op:"task"}` for action items. Uncertain ‚Üí `hypothesis/candidate`, never self-promoted.
- **`daily-command-center`** *(modify)* ‚Äî add `state/` as a source. Promoted records shown as
  trusted; candidates/hypotheses shown *as* candidates. Today-only filter unchanged.
- **`watchdog`** *(new)* ‚Äî reads `index.yaml` + each routine's `memory.md`. Checks SLA
  freshness, schema validity, orphan `supersedes`, count reconciliation. Output: health report
  + `{op:"reindex"}` proposals.
- **`weekly-review`** *(repurpose)* ‚Äî from Wins/Metrics into the **curator**: reads full
  `state/`, proposes `transition`/`correct` ops with a `reason`. Nothing auto-applies.

## Testing

- `core/state.ts` ‚Äî unit tests: add stamps id/created; in-place transition; correct
  (new supersedes old, old archived); per-kind status validation rejects `decision/candidate`;
  deterministic `decisions.md` projection; `index.yaml` count recompute.
- `apply` ‚Äî `--dry-run` writes nothing; invalid proposal refused with visible failure;
  `task` op routes to Linear (mocked), not JSONL; writes confined to `state/`.
- Each routine ‚Äî `response.ok.json` (valid proposals) + `response.bad.json` (trips schema)
  + a `.test.ts` via `MockAdapter`. No network in tests.

## Out of scope

- Live connector wiring (Granola/Linear runtime credentials) ‚Äî a host concern.
- Auto-commit of `apply` results ‚Äî every write stays a manual, reviewable git change.
- Event-sourcing of `learnings.jsonl` ‚Äî explicitly rejected in favour of mutable + git.
